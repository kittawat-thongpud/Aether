Aether-Core: Engineering & Programming RulesMission-Critical Standards for High-Performance MiddlewareThese rules are derived from NASA’s "Power of 10" safety-critical rules and Tesla-grade professional engineering standards, adapted specifically for the Aether-Core data-oriented architecture.1. Control Flow & Determinism (NASA Rule 1, 2)Simple Control Flow Only: Avoid complex flow constructs. No recursion (direct or indirect) is allowed to ensure stack safety. All control flow must be predictable and verifiable by static analysis tools.Fixed Loop Bounds: All loops must have a fixed, verifiable upper limit. If a loop is processing a stream of sensor data, it must have a timeout or a maximum iteration count to prevent infinite execution and Jitter.2. Memory Management (NASA Rule 3)No Dynamic Allocation After Initialization: All memory for Shared Memory (SHM) pools, Schema Registries, and DMA buffers must be allocated during the Init phase. Use of malloc, free, or Rust’s Box/Vec resizing in the Fast Path (Runtime) is strictly forbidden to prevent non-deterministic latency.Zero-Copy Enforcement: Any function in the Data Plane that performs a memcpy of payload data must be explicitly justified and documented. Prefer pointer-wrapping (Zero-copy) for all cross-language transitions.3. Data Integrity & Schema SafetyMandatory UUID Verification: No data block shall be accessed from SHM without first verifying its UUID. If the UUID does not match the local Schema, the "Drop if Unknown" policy must be executed immediately.Atomic SHM Updates: All updates to the Schema Registry or Shared Memory status flags must use Atomic operations (e.g., AtomicCAS). Never use raw pointers for shared state variables without memory barriers or mutexes.4. Function Design & Scope (NASA Rule 4, 6)Single-Screen Functions: Functions must be concise. A function should ideally not exceed 60 lines of code. If a function cannot be understood at a glance, it is doing too much and must be decomposed.Strict Data Scoping: The scope of data objects must be as small as possible. Global variables are prohibited unless they are part of the pre-allocated Data Plane Registry. All access to SHM must be through the Aether-Core API to ensure visibility rules are respected.5. Defensive Programming (NASA Rule 5, 7)High Assert Density: Every function must contain at least two assert statements (or panic! in Rust) to check for impossible conditions, such as Null pointers after mapping or UUID format violations.Validate Every Return Value: Every system call, Zenoh API call, and SHM mapping call must have its return value checked. If a call can fail, the failure must be handled; it is never acceptable to ignore a result.6. Hardware & Proxy Rules (Tesla/Professional Engineering)DMA Buffer Alignment: MCU (STM32) buffers intended for Host-Proxy transfer must use #pragma pack(1) or __attribute__((packed)) and be 32-bit aligned to ensure the Host reads them without realignment overhead.Host-Proxy Isolation: The MCU Proxy on the Host must validate the size of incoming DMA packets before writing to SHM to prevent buffer overflows into adjacent memory pools.7. Reliability & TroubleshootingStructured Logging with Context: Logs must include the UUID, Topic name, and Timestamp. Use "Operational Logs" for system status and "Debug Logs" for byte-level inspection.Watchdog Integrity: A background "Watchdog" thread must monitor the health of the Schema Registry. If a Registry Mutex is held longer than the defined timeout (Deadlock), the Watchdog must trigger a system-wide RE-INIT signal.8. Development WorkflowCompile with All Warnings: All code must compile with zero warnings using the strictest flags (e.g., -Wall -Wextra -Werror for C/C++, #![deny(warnings)] for Rust).Declarative Maps over Conditionals: Replace large switch/case or if/else logic for Schema handling with declarative Hash Maps or V-Tables to ensure O(1) lookup time.Definition of Done for Aether-Core[ ] Code is 100% warning-free.[ ] Zero dynamic allocation in the Fast Path.[ ] Every Topic access is guarded by UUID validation.[ ] Unit tests cover the "Drop if Unknown" edge cases.[ ] Memory alignment is verified for MCU-to-Host transfer."Write for the next engineer, optimize for the hardware, and never trust a packet without a UUID."
